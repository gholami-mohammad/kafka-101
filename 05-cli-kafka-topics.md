## First thing First

All CLI examples will be executed on Kafka on localhost with no security enabled. If you want to connect to Kafka with sasl security enabled, you need to create a config file like below and pass it to the command:

```
#kafka.config

security.protocol=SASL_SSL
sasl.jaas.config=org.apache.kafka.common.security.plain.PlainLoginModule required username="YOUR_USERNAME" password="YOUR_PASSWORD";
sasl.mechanism=PLAIN
```

And when running any command, you can pass the config file using `--command-config` flag. For example:

```sh
kafka-topics --command-config kafka.config --bootstrap-server my.kafka.cluster.example.com:9092 --list
```

## Create Kafka Topics

**NOTE:** AVOID having "\_" and "." both at same time in the topic name.

Create topic with default partitions length:

```sh
kafka-topics --bootstrap-server localhost:9092 --create --topic first-topic
```

Create topic with preferred partitions length:

```sh
kafka-topics --bootstrap-server localhost:9092 --create --topic second-topic --partitions 5
```

Create topic with replication factor:

```sh
kafka-topics --bootstrap-server localhost:9092 --create --topic third-topic --partitions 5 --replication-factor 1
```

Replication factor can have max value of your cluster available brokers number. In the other word, if your cluster has 3 brokers, this value can be 3 in maximum.

## List Kafka Topics

```sh
kafka-topics --bootstrap-server localhost:9092 --list
```

## Describe Kafka Topics

```sh
kafka-topics --bootstrap-server localhost:9092 --topic third-topic --describe
```

Command output is something like this:

```
Topic: third-topic      TopicId: IBUwWLU-Qei8xShWEPelJg PartitionCount: 5       ReplicationFactor: 1    Configs: min.insync.replicas=1,segment.bytes=1073741824
        Topic: third-topic      Partition: 0    Leader: 1       Replicas: 1     Isr: 1  Elr:    LastKnownElr:
        Topic: third-topic      Partition: 1    Leader: 1       Replicas: 1     Isr: 1  Elr:    LastKnownElr:
        Topic: third-topic      Partition: 2    Leader: 1       Replicas: 1     Isr: 1  Elr:    LastKnownElr:
        Topic: third-topic      Partition: 3    Leader: 1       Replicas: 1     Isr: 1  Elr:    LastKnownElr:
        Topic: third-topic      Partition: 4    Leader: 1       Replicas: 1     Isr: 1  Elr:    LastKnownElr:
```

What do we have:

First Line: is topic information and it's clear.

Next 5 lines: we have 5 partition for the `third-topic`. So we have 5 lines here and each line, describing one partition.
This line is the standard output generated by the `kafka-topics.sh --describe` command. It provides a detailed, partition-level view of a topic's health and distribution across the Kafka cluster.

Here is the breakdown of what each part means based on our specific output for the 2nd line:

- **`Topic: third-topic`**
    - **Meaning:** This is simply the name of the Kafka topic being described.

- **`Partition: 0`**
    - **Meaning:** Kafka topics are split into one or more partitions for scalability. This line is specifically describing the status of Partition `0` (partitions are zero-indexed, meaning the first partition is always 0).

- **`Leader: 1`**
    - **Meaning:** This indicates the Broker ID (in this case, Broker `1`) that is currently the **Leader** for this specific partition.
    - **Significance:** The leader handles all read and write requests for this partition. If a producer wants to send a message to Partition 0, or a consumer wants to read from it, they must connect to Broker 1.

- **`Replicas: 1`**
    - **Meaning:** This is a list of all Broker IDs that are configured to store a copy (replica) of this partition, regardless of whether those brokers are currently online or offline.
    - **Significance:** In this example, the list only contains broker `1`. This means this topic has a Replication Factor of 1. Data is only stored on broker 1; there are no backups on other brokers.

- **`Isr: 1`**
    - **Meaning:** **In-Sync Replicas (ISR)**. This is a subset of the `Replicas` list. It shows the Broker IDs that are currently alive, fully caught up with the Leader, and actively replicating data.
    - **Significance:** If the current leader fails, only a broker in the ISR list can be automatically elected as the new leader without risking data loss. Since our Replicas list only contains broker `1`, the ISR can only be `1`.

- **`Elr:`**
    - **Meaning:** **Eligible Leader Replicas (ELR)**. This is a newer feature in Kafka (introduced in KIP-966 for Kafka 3.7+).
    - **Significance:** ELR bridges the gap between strict strict ISR and completely random "unclean leader election". If all ISRs fail, Kafka usually halts to prevent data loss. The ELR list contains replicas that are _not_ currently in the strict ISR list, but are deemed "close enough" or safe enough to be elected as a leader if you want to prioritize availability over strict consistency, without resorting to the worst-case scenario of picking a totally out-of-sync replica.
    - **In our output:** It is blank because either the cluster is perfectly healthy (ISR is sufficient), or the feature isn't actively managing degraded replicas for this single-replica partition.

- **`LastKnownElr:`**
    - **Meaning:** **Last Known Eligible Leader Replicas**. Also part of the KIP-966 update.
    - **Significance:** If the current ELR list becomes empty (perhaps those brokers also went offline), this field remembers the last known good set of Eligible Leader Replicas. It helps the cluster make the smartest possible recovery decision if a catastrophic failure takes out both the Leader, the ISR, and the current ELR.
    - **In your output:** Blank, for the same reasons as ELR.

## Delete a Kafka Topic

```sh
kafka-topics --bootstrap-server localhost:9092 --topic first-topic --delete
```
