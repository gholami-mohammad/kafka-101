# دوره «Designing Events and Event Streams» آدام بلمار

ایونت ثبت غیرقابل تغییری از چیزی است که در کسب‌وکار رخ داده. تفاوت مهمی بین داده‌های داخلی سرویس و داده‌هایی که به بیرون اشتراک گذاشته می‌شوند وجود دارد. طراحی خوب ایونت‌ها بر چهار بعد اصلی استوار است که در ادامه به تفصیل توضیح داده شده.

## بعد اول: Fact vs. Delta Event Types

مفهوم Fact Event (یا State Event): وضعیت کامل و خودکفای یک موجودیت را در لحظهٔ خاص نشان می‌دهد. مصرف‌کننده برای درک وضعیت نیازی به هیچ اطلاعات قبلی ندارد.

مفهوم Delta Event: فقط تغییرات ایجاد شده را ثبت می‌کند. حجم کمتری دارد، اما مصرف‌کننده باید حالت‌های قبلی را نگه دارد تا بتواند وضعیت کامل را بازسازی کند.

### مثال واقعی: سیستم سفارشات (Order)

فرض کنید یک سفارش با شناسه `ORD-1234` داریم.

#### ۱. استفاده از Fact Event (State Event)

وقتی وضعیت سفارش تغییر می‌کند، یک ایونت کامل می‌فرستیم:

```json
{
	"eventType": "OrderStateChanged",
	"orderId": "ORD-1234",
	"timestamp": "2026-05-26T10:15:30Z",
	"customer": {
		"id": "CUST-987",
		"name": "علی رضایی",
		"phone": "09123456789"
	},
	"items": [
		{
			"productId": "P1",
			"name": "لپ‌تاپ",
			"quantity": 1,
			"price": 45000000
		},
		{ "productId": "P2", "name": "موس", "quantity": 2, "price": 1200000 }
	],
	"totalAmount": 47400000,
	"status": "Shipped",
	"shippingAddress": "تهران، خیابان ولیعصر...",
	"paymentStatus": "Paid",
	"trackingCode": "TRK-556677"
}
```

**مزیت:** هر کسی این ایونت را ببیند، دقیقاً می‌فهمد سفارش الان در چه وضعیتی است. نیازی به خواندن ایونت‌های قبلی ندارد.

#### ۲. حالت Delta Event (تغییرات)

برای همان سفارش، ایونت‌های Delta به این شکل هستند:

**ایونت اول (ایجاد سفارش):**

```json
{
  "eventType": "OrderCreated",
  "orderId": "ORD-1234",
  "timestamp": "2026-05-25T08:20:00Z",
  "changes": {
    "status": "Created",
    "totalAmount": 47400000,
    "itemsAdded": [ ... ]
  }
}
```

**ایونت دوم (پرداخت):**

```json
{
	"eventType": "OrderUpdated",
	"orderId": "ORD-1234",
	"timestamp": "2026-05-25T08:25:10Z",
	"changes": {
		"paymentStatus": "Paid",
		"status": "Processing"
	}
}
```

**ایونت سوم (ارسال):**

```json
{
	"eventType": "OrderUpdated",
	"orderId": "ORD-1234",
	"timestamp": "2026-05-26T10:15:30Z",
	"changes": {
		"status": "Shipped",
		"trackingCode": "TRK-556677"
	}
}
```

**معایب Delta:** مصرف‌کننده باید همه این ایونت‌ها را به ترتیب بخواند و روی حالت اولیه اعمال کند تا وضعیت نهایی را بفهمد.

---

### مقایسه عملی

| مورد               | Fact Event (State)            | Delta Event                     |
| ------------------ | ----------------------------- | ------------------------------- |
| حجم ایونت          | بزرگ‌تر                       | خیلی کوچک‌تر                    |
| پیچیدگی مصرف‌کننده | خیلی ساده                     | پیچیده (نیاز به نگهداری حالت)   |
| بازسازی وضعیت      | فوری و آسان                   | نیاز به پردازش همه تغییرات قبلی |
| مناسب برای         | داشبورد، میکروسرویس‌های مستقل | سیستم‌هایی با تغییرات خیلی زیاد |
| فضای ذخیره‌سازی    | بیشتر                         | خیلی کمتر                       |

---

**نکته کلیدی:**
معمولاً در سیستم‌های مدرن ترکیبی استفاده می‌شود.

## بعد دوم: Modeling as Normalized vs. Denormalized

این بعد یکی از مهم‌ترین تصمیم‌گیری‌ها در طراحی ایونت‌هاست. اینجا باید انتخاب کنیم که **داده‌های مرتبط** رو چطور مدیریت کنیم.

### ۱. نرمال‌سازی - Normalized Modeling

در این رویکرد، ایونت فقط شامل اطلاعات اصلی و مستقیم یک موجودیت (Entity) است و داده‌های مرتبط را داخل خودش نمی‌آورد.

**ویژگی‌ها:**

- هر موجودیت جریان (Stream) جداگانه خودش را دارد.
- اگر نیاز به اطلاعات مرتبط باشد، مصرف‌کننده باید چندین جریان را با هم **جوین** کند (مثل جوین جدول‌ها در دیتابیس).
- حجم هر ایونت کوچک‌تر است چون فقط داده‌های ضروری را دارد.
- داده‌ها فقط یک بار ذخیره می‌شوند (بدون تکرار).

**مزایا:**

- تغییر در یک موجودیت فقط روی جریان خودش تأثیر می‌گذارد.
- جلوگیری از تکرار داده (Data Duplication).
- حجم ذخیره‌سازی و انتقال داده کمتر.
- مناسب وقتی موجودیت‌های مرتبط خیلی زیاد تغییر می‌کنند.

**معایب:**

- مصرف‌کننده پیچیده‌تر می‌شود (نیاز به مدیریت جوین، state و همگام‌سازی).
- تأخیر احتمالی در دسترسی به تصویر کامل (چون باید جوین انجام شود).
- پیچیدگی بیشتر در سمت مصرف‌کننده برای بازسازی وضعیت کامل.

### ۲. دنورمال‌سازی - Denormalized Modeling

در این رویکرد، تمام داده‌های مورد نیاز و مرتبط را **داخل خود ایونت** قرار می‌دهیم. یعنی مصرف‌کننده همه‌چیز را در یک پیام دریافت می‌کند و نیازی به جوین ندارد.

**ویژگی‌ها:**

- ایونت‌ها غنی (Rich) و خودکفا هستند.
- داده‌های مرتبط از جداول یا جریان‌های دیگر داخل payload ایونت کپی می‌شوند.
- مصرف‌کننده خیلی ساده‌تر کار می‌کند.

**مزایا:**

- مصرف‌کننده بسیار ساده است (فقط یک جریان را بخواند و همه اطلاعات را داشته باشد).
- سرعت پردازش بالاتر برای مصرف‌کننده (بدون نیاز به جوین در زمان واقعی).
- مناسب برای تیم‌هایی که مصرف‌کننده‌های متنوع و ساده می‌خواهند (مثل تیم‌های تحلیلی یا میکروسرویس‌های مختلف).

**معایب:**

- حجم ایونت‌ها خیلی بزرگ‌تر می‌شود.
- تکرار داده (Duplication) به وجود می‌آید.
- اگر داده مرتبط تغییر کند، باید ایونت‌های قبلی را هم به‌روزرسانی کنیم یا ایونت‌های جدید جبرانی بفرستیم (که پیچیده است).
- نگهداری و تکامل سخت‌تر در بلندمدت.

---

### متدهای رایج برای پیاده‌سازی Denormalized

چند روش عملی برای دنورمال کردن ایونت‌ها وجود دارد:

روش اول: **Embedding مستقیم (Direct Embedding)**:

تمام داده‌های مرتبط را مستقیماً داخل ساختار JSON/Avro ایونت قرار می‌دهیم.  
مثال: در ایونت `OrderCreated`، علاوه بر اطلاعات سفارش، اطلاعات کامل مشتری (نام، آدرس، سطح وفاداری) و جزئیات محصولات را هم داخل آن می‌گذاریم.

روش دوم: **Enrichment در سمت Producer**:

قبل از ارسال ایونت به Kafka، سرویس Producer داده‌های مرتبط را از سرویس‌های دیگر یا دیتابیس می‌گیرد و داخل ایونت غنی‌سازی (Enrich) می‌کند.  
این روش خیلی رایج است.

روش سوم: **Stream Enrichment با ksqlDB یا Flink**:

از ابزارهای استریم‌پردازش برای جوین کردن جریان‌ها در سمت تولیدکننده استفاده می‌کنیم و بعد ایونت دنورمال‌شده را منتشر می‌کنیم.

روش چهارم: **Using Materialized Views**:

یک نمای مادی‌سازی‌شده (Materialized View) از داده‌ها نگه می‌داریم و هر بار که ایونت اصلی منتشر می‌شود، از این نمای به‌روز برای ساخت ایونت کامل استفاده می‌کنیم.

روش پنجم: **Event Versioning با Snapshot کامل**:

به صورت دوره‌ای یا در زمان تغییرات مهم، یک ایونت Snapshot کامل و دنورمال‌شده منتشر کنیم تا مصرف‌کننده‌ها بتوانند وضعیت کامل را یکجا دریافت کنند.

---

**نکته مهم در انتخاب:**

- اگر مصرف‌کننده‌ها متنوع هستند و سادگی برایشان خیلی مهم است → **Denormalized** بهتر است.
- اگر حجم داده زیاد است، تغییرات مرتبط زیاد هستند و تیم فنی قوی دارید → **Normalized** منطقی‌تر است.
- بسیاری از سیستم‌های واقعی ترکیبی (Hybrid) استفاده می‌کنند: برخی جریان‌ها نرمال و برخی دنورمال.

---

### استفاده از Outbox Pattern و Change Data Capture برای Denormalize کردن داده

در ادامه بحث Normalized و Denormalized، یکی از بخش‌های مهم این بود که وقتی منبع داده ما یک دیتابیس رابطه‌ای (Relational Database) است، چطور ایونت‌های **دنورمال‌شده** بسازیم. اینجا دو الگوی خیلی کاربردی معرفی شد:

#### ۱. روش Change Data Capture (CDC)

اصطلاح CDC یعنی گرفتن تغییرات مستقیم از دیتابیس و تبدیل آن‌ها به ایونت.

**چگونگی کار برای Denormalization:**

- با ابزارهایی مثل Debezium یا CDC Connectorهای Confluent، تغییرات جداول دیتابیس را capture می‌کنیم.
- چون جداول معمولاً نرمال‌شده هستند (Normalized)، خروجی اولیه CDC هم نرمال است (فقط تغییرات یک جدول).
- برای دنورمال کردن، باید در لایه بعد (معمولاً با ksqlDB، Flink یا یک سرویس جدا) جریان‌های مختلف را جوین کنیم و ایونت غنی‌شده بسازیم.

**مزایا:**

- بدون نیاز به تغییر کد اپلیکیشن موجود.
- تغییرات را به‌صورت واقعی (Real-time) capture می‌کند.

**معایب:**

- دنورمال‌سازی باید در لایه بیرون از دیتابیس انجام شود.
- ممکن است در ابتدا ایونت‌ها خیلی نرمال بمانند و مصرف‌کننده مجبور به جوین شود.
- مدیریت ترتیب تغییرات و transactional consistency سخت‌تر است.

#### ۲. روش Transactional Outbox Pattern

این الگو یکی از بهترین روش‌ها برای تولید ایونت‌های دنورمال‌شده با اطمینان بالا است.

**چگونگی کار:**

- وقتی یک تراکنش در دیتابیس انجام می‌شود (مثلاً ایجاد سفارش + به‌روزرسانی موجودی کالا)، همزمان در همان تراکنش، یک رکورد در جدول **Outbox** می‌نویسیم.
- این رکورد شامل **اطلاعات کامل و دنورمال‌شده** است که می‌خواهیم به صورت ایونت منتشر شود (نه فقط کلید خارجی).
- یک فرآیند جدا (Outbox Processor یا CDC روی جدول Outbox) رکوردهای جدول Outbox را می‌خواند و به Kafka منتشر می‌کند.
- بعد از انتشار موفق، رکورد Outbox پاک یا علامت‌گذاری می‌شود.

**مزایای مهم برای Denormalization:**

- دنورمال‌سازی دقیقاً در لحظه نوشتن تراکنش انجام می‌شود (داخل اپلیکیشن Producer).
- تضمین Atomicity: یا هم داده در دیتابیس ذخیره می‌شود و هم ایونت در Outbox نوشته می‌شود.
- مصرف‌کننده ایونت کامل و خودکفا دریافت می‌کند (بدون نیاز به جوین).
- از مشکل Dual Write جلوگیری می‌کند.

**نکات عملی:**

- جدول Outbox باید سبک باشد و رکوردهای قدیمی به‌موقع پاک شوند.
- می‌توان از CDC روی جدول Outbox استفاده کرد تا انتشار اتوماتیک شود.
- این الگو برای سیستم‌هایی که consistency خیلی مهم است (مثل سفارش، پرداخت، رزرو) عالی عمل می‌کند.

---

**مقایسه کوتاه:**

- روش **CDC ساده**: سریع‌تر راه می‌افتد اما دنورمال‌سازی را به بعد موکول می‌کند.
- روش **Outbox Pattern**: کمی پیچیده‌تر در پیاده‌سازی، اما ایونت‌های باکیفیت‌تر و دنورمال‌شده‌تری تولید می‌کند و consistency بالاتری دارد.

بسیاری از تیم‌ها ترکیبی استفاده می‌کنند: از Outbox برای تراکنش‌های مهم و CDC برای تغییرات ساده‌تر.

## بعد سوم: Modeling as Single vs. Multiple Event Streams

این بعد درباره این تصمیم مهم است که **چگونه ایونت‌های مختلف را در جریان‌ها (Topicها) سازماندهی کنیم**.

### تعریف دو رویکرد

**۱. رویکرد Single Event Stream (یک نوع ایونت در هر جریان)**  
هر نوع ایونت، جریان (Topic) کاملاً جداگانه خودش را دارد.

**۲. رویکرد Multiple Event Streams (چند نوع ایونت در یک جریان)**  
چند نوع ایونت مختلف را داخل یک Topic واحد قرار می‌دهیم.

---

### ۱. رویکرد Single Event Stream (توصیه‌شده در بیشتر موارد)

**ویژگی‌ها:**

- هر Topic فقط یک نوع ایونت خاص را حمل می‌کند (مثلاً `OrderCreated`، `OrderShipped`، `OrderCancelled` هر کدام Topic جدا).
- نام Topic معمولاً واضح و معنادار است: `orders.created.v1` یا `order-events.created`.

**مزایا:**

- **کشف و درک آسان**: وقتی کسی Topic را می‌بیند، دقیقاً می‌داند چه ایونت‌هایی داخل آن است.
- **نسخه بندی یا Versioning ساده**: می‌توانی راحت نسخه‌های مختلف یک ایونت را مدیریت کنی (مثلاً `orders.created.v1` و `orders.created.v2`).
- **مصرف انتخابی**: مصرف‌کننده‌ها دقیقاً همان ایونتی را که نیاز دارند subscribe کنند، بدون دریافت نویز اضافی.
- **امنیت و مجوزدهی**: کنترل دسترسی به سطح Topic راحت‌تر است.
- **مانیتورینگ و مقیاس‌پذیری**: راحت‌تر می‌توانی مصرف هر نوع ایونت را جداگانه نظارت کنی.

**معایب:**

- تعداد Topicها زیاد می‌شود (ممکن است مدیریت سخت شود).
- اگر تعداد انواع ایونت‌ها خیلی زیاد باشد، overhead عملیاتی افزایش پیدا می‌کند.

---

### ۲. رویکرد Multiple Event Streams (چند نوع در یک جریان)

**ویژگی‌ها:**

- چندین نوع ایونت مرتبط را داخل یک Topic واحد می‌گذاریم (مثلاً همه رویدادهای سفارش در Topic `order-events`).
- معمولاً از یک هدر (Header) یا فیلد `eventType` داخل payload برای تشخیص نوع ایونت استفاده می‌شود.

**مزایا:**

- تعداد Topicها کمتر می‌شود (مدیریت ساده‌تر).
- ایونت‌های مرتبط در یک جا جمع هستند (مخصوصاً وقتی workflowها به هم مرتبط‌اند).
- مصرف‌کننده‌هایی که به چندین نوع ایونت نیاز دارند، فقط یک Topic را بخوانند.
- در برخی موارد مصرف منابع کمتر (کمتر Partition).

**معایب:**

- **کشف سخت‌تر**: مصرف‌کننده باید بداند چه نوع ایونت‌هایی داخل این Topic وجود دارد.
- **نسخه بندی یا Versioning پیچیده**: مدیریت نسخه‌های مختلف انواع ایونت‌ها داخل یک Topic سخت‌تر است.
- **نویز**: مصرف‌کننده ممکن است ایونت‌هایی دریافت کند که به آن‌ها نیاز ندارد و مجبور به فیلتر کردن شود.
- **امنیت**: کنترل دسترسی granular (جزئی) سخت‌تر است.

---

### مثال واقعی: سیستم سفارشات

**رویکرد Single Event Stream:**

- `orders.created.v1`
- `orders.updated.v1`
- `orders.shipped.v1`
- `orders.cancelled.v1`
- `orders.delivered.v1`

**رویکرد Multiple Event Streams:**

- یک Topic به نام `order-events` که شامل همه انواع بالا است و هر ایونت فیلد `eventType` دارد.

---

**بهترین شیوه پیشنهادی:**
در بیشتر پروژه‌های واقعی، رویکرد **Single Event Stream** (هر نوع ایونت در Topic جدا) ترجیح داده می‌شود، چون سیستم را در بلندمدت قابل نگهداری‌تر و قابل درک‌تر نگه می‌دارد.  
فقط در مواردی که ایونت‌ها خیلی به هم وابسته و کم‌تعداد هستند، از Multiple Event Streams استفاده می‌شود.

**نکته مهم:**  
حتی اگر از Multiple استفاده کردی، حتماً از فیلد `eventType` و Headerهای مناسب استفاده کن تا مصرف‌کننده‌ها بتوانند راحت فیلتر کنند.

---

### بعد چهارم: Modeling As Discrete vs. Continuous Event Flows

این بعد درباره **ماهیت زمانی و طول عمر** جریان‌های ایونت است. یعنی اینکه جریان ایونت ما قرار است **محدود و پایان‌دار** باشد یا **پیوسته و نامحدود**.

### ۱. جریان‌های گسسته / پایان‌دار یا Discrete Event Flows

این نوع جریان‌ها شروع و پایان مشخصی دارند و معمولاً با یک فرآیند کسب‌وکار خاص و محدود مرتبط هستند.

**ویژگی‌های اصلی:**

- جریان دارای طول عمر مشخص است (شروع -> مراحل میانی -> پایان).
- تعداد ایونت‌ها در یک workflow محدود است.
- پس از رسیدن به نتیجه نهایی، جریان عملاً تمام می‌شود.
- معمولاً با مفهوم **Workflow** یا **Business Process** هم‌خوانی دارد.

**مثال‌های واقعی:**

- فرآیند ثبت سفارش: `OrderCreated → OrderPaid → OrderShipped → OrderDelivered → OrderCompleted`
- فرآیند رزرو هتل: `ReservationCreated → PaymentProcessed → CheckIn → CheckOut`
- فرآیند ثبت‌نام کاربر: `UserRegistered → EmailVerified → ProfileCompleted`

**مزایا:**

- درک کسب‌وکار و traceability بسیار آسان است.
- پیاده‌سازی State Machine یا Saga Pattern راحت‌تر است.
- نظارت و مانیتورینگ فرآیند (مثل تأخیر در مراحل) ساده‌تر است.
- مصرف‌کننده‌ها می‌توانند راحت‌تر منطق "اتمام فرآیند" را پیاده کنند.

**معایب:**

- اگر فرآیند خیلی طولانی شود، مدیریت حالت (State) سخت می‌شود.
- نیاز به مکانیزم‌هایی مثل Timeout یا Cancellation دارد.

### ۲. جریان‌های پیوسته / بدون پایان یا Continuous Event Flows

این نوع جریان‌ها هیچ پایان مشخصی ندارند و به صورت مداوم ایونت تولید می‌کنند.

**ویژگی‌های اصلی:**

- ایونت‌ها به صورت نامحدود و پیوسته تولید می‌شوند.
- معمولاً برای نظارت، تحلیل و تصمیم‌گیری استفاده می‌شوند.
- فاقد مفهوم "اتمام فرآیند" هستند.

**مثال‌های واقعی:**

- کلیک‌ها و رفتار کاربران در وبسایت (UserClickStream)
- داده‌های سنسورهای IoT (دما، رطوبت، موقعیت)
- متریک‌های فنی سیستم (CPU Usage, API Latency, Error Rate)
- تراکنش‌های مالی لحظه‌ای در یک صرافی ارز دیجیتال
- لاگ فعالیت کاربران در اپلیکیشن

**مزایا:**

- عالی برای نظارت، تشخیص الگو و تحلیل‌های آماری.
- مناسب برای سیستم‌های مبتنی بر Machine Learning و تشخیص تقلب.
- مقیاس‌پذیری افقی راحت‌تر.

**معایب:**

- حجم داده بسیار بالا می‌رود.
- نگهداری حالت (State) طولانی‌مدت پیچیده و پرهزینه است.
- مصرف‌کننده باید بتواند با جریان‌های بسیار بزرگ و بدون پایان کنار بیاید.

---

### مقایسه خلاصه

| معیار               | Discrete Event Flows          | Continuous Event Flows          |
| ------------------- | ----------------------------- | ------------------------------- |
| طول عمر جریان       | محدود و دارای پایان           | نامحدود و پیوسته                |
| نوع استفاده         | فرآیندهای کسب‌وکار (Workflow) | نظارت، تحلیل، metrics           |
| مدیریت حالت (State) | نسبتاً ساده                   | پیچیده و پرهزینه                |
| حجم داده            | معمولاً متوسط                 | بسیار بالا                      |
| مثال                | سفارش، پرداخت، رزرو           | کلیک کاربر، سنسور، لاگ سیستم    |
| الگوهای مناسب       | Saga, State Machine           | Real-time Analytics, Monitoring |

---

**نکته کلیدی در طراحی:**

- رویکرد **Discrete**: تمرکز روی **ترتیب، کامل بودن و اتمام** فرآیند است. اغلب از Fact Eventها استفاده می‌شود.
- رویکرد **Continuous**: تمرکز روی **سرعت، حجم و تشخیص الگو** است. اغلب از Delta Eventها استفاده می‌شود.

در سیستم‌های واقعی معمولاً هر دو نوع جریان وجود دارد. مثلاً در یک فروشگاه آنلاین:

- جریان سفارشات → **Discrete**
- جریان کلیک‌ها و پیشنهادات محصول → **Continuous**

## بهترین شیوه های نامگذاری، شناسه ها، متادیتا و هدر ها

**یادداشت‌های طراحی ایونت – بهترین شیوه‌ها (Best Practices): نام‌گذاری، شناسه‌ها، متادیتا و هدرها**

این بخش یکی از کاربردی‌ترین قسمت‌های طراحی ایونت است. جزئیات به ظاهر کوچک که تأثیر بسیار زیادی بر قابلیت نگهداری، دیباگینگ، امنیت و تکامل سیستم در بلندمدت دارند.

### ۱. Naming Conventions (بهترین شیوه‌های نام‌گذاری)

**الف) نام‌گذاری Topicها**

- فرمت پیشنهادی: `domain.event-type.version`
- مثال:
    - `orders.created.v1`
    - `orders.shipped.v2`
    - `users.registered.v1`
    - `payments.refunded.v1`

**ب) نام‌گذاری نوع ایونت (Event Type)**

- از **PascalCase** استفاده کنید: `OrderCreated`, `OrderShipped`, `UserRegistered`
- نام‌ها باید واضح، خاص و معنادار باشند.
- از کلمات کلی و مبهم مثل `Updated` یا `Changed` به تنهایی اجتناب کنید. بهتر است بنویسید `OrderStatusUpdated` یا `OrderItemsUpdated`.

**نکات مهم نام‌گذاری:**

- از گذشته ساده (Past Tense) استفاده کنید چون ایونت چیزی است که **اتفاق افتاده**.
- نسخه‌بندی (Versioning) را فراموش نکنید.
- از Prefix دامنه (Domain) برای گروه‌بندی منطقی Topicها استفاده کنید.

### ۲. شناسه‌ها (IDs)

هر ایونت باید دارای شناسه‌های مناسب باشد:

یک: شناسه خود ایونت : **Event ID**:

۱- همیشه یک **UUID v4** (یا KSUID/ULID) باشد.

۲- هدف: یکتایی کامل (Uniqueness)، Deduplication و Traceability.

دو: شناسه موجودیت اصلی: **Entity ID**:
۱- مثلاً `orderId`, `userId`, `paymentId`.

۲- این شناسه معمولاً Business Key است.

سه: شناسه **Correlation ID**:

۱- برای ردیابی یک درخواست یا workflow کامل در کل سیستم استفاده می‌شود.

۲- همه ایونت‌هایی که به یک عملیات اولیه مربوط هستند، این شناسه را دارند.

سه: شناسه **Causation ID**:

شناسه ایونتی که باعث ایجاد این ایونت شده است (برای ردیابی زنجیره علت و معلول).

**مثال ساختار:**

```json
{
  "eventId": "123e4567-e89b-12d3-a456-426614174000",
  "orderId": "ORD-98765",
  "correlationId": "corr-abc123",
  "causationId": "evt-456789",
  ...
}
```

### ۳. متادیتا و هدر ها Metadata & Headers

یکی از مهم‌ترین توصیه‌ها این است که **اطلاعات زمینه‌ای (Context)** را داخل بدنه اصلی ایونت (Payload) نگذارید، بلکه از **Headers** یا یک فیلد `metadata` جدا استفاده کنید.

**اطلاعات پیشنهادی برای Headers / Metadata:**

| دسته‌بندی     | فیلدهای پیشنهادی                     | توضیح                              |
| ------------- | ------------------------------------ | ---------------------------------- |
| منبع و مبدا   | `source`, `service`, `instanceId`    | کدام سرویس این ایونت را تولید کرده |
| کاربر و امنیت | `userId`, `tenantId`, `roles`        | برای authorization و multi-tenancy |
| ردیابی        | `traceId`, `spanId`, `correlationId` | برای Distributed Tracing           |
| زمانی         | `producedAt`, `eventTime`            | زمان تولید واقعی ایونت             |
| محیط          | `environment` (dev/staging/prod)     | تشخیص محیط                         |
| نسخه          | `schemaVersion`, `eventVersion`      | مدیریت نسخه schema                 |
| فنی           | `contentType`, `compression`         | نوع محتوا و فشرده‌سازی             |

**مثال عملی Headers در Kafka:**

- `eventId`: 123e4567-...
- `eventType`: OrderShipped
- `source`: order-service
- `userId`: U-456
- `correlationId`: corr-xyz789
- `producedAt`: 2026-05-26T12:22:15Z
- `environment`: production

### توصیه‌های کلی بهترین شیوه‌ها

- مبحث **Schema Evolution** را جدی بگیرید (Backward, Forward, Full Compatibility).
- همیشه `eventId` + `eventType` + `producedAt` را داشته باشید.
- همواره Payload را تمیز و متمرکز نگه دارید (Business Data) و Context را به Headers ببرید.
- از نام‌گذاری یکنواخت و استاندارد در کل سازمان استفاده کنید (بهتر است یک Event Catalog یا Registry داشته باشید).
- فیلد Headers را برای اطلاعاتی که مصرف‌کننده‌های مختلف ممکن است به آن نیاز داشته باشند، استفاده کنید.
